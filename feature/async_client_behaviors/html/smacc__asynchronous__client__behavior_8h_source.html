13  // Asnchronous client behaviors are used when the onEntry or onExit function execution is slow
14  // CONCEPT: this funcionality is related with the orthogonality of SmaccState machines.
15  // No behavior should block the creation of other behaviors, all of them conceptually start in parallel.
16  // Alternative for long duration behaviors: using default-synchromous SmaccClientBehaviors with the update method
17  // ASYNCHRONOUS STATE MACHINES DESIGN NOTES: Asynchromous behaviors can safely post events and use its local methods,
18  // but the interaction with other components or elements of
19  // the state machine is not by-default thread safe and must be manually implemented. For example, if some element of the architecture
20  // (components, states, clients) need to access to this behavior client information it is needed to implement a mutex for the internal
21  // state of this behavior. Other example: if this behavior access to some component located in other thread, it is also may be needed
