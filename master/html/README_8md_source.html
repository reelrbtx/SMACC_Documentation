17 # <img src="http://smacc.ninja/wp-content/uploads/2019/07/SMACC-Logo-Pixelate-4-copy.png" width="30" align="left"/> SMACC
18 
19 SMACC is an event-driven, asynchronous, behavioral state machine library for real-time ROS (Robotic Operating System) applications written in C++, designed to allow programmers to build robot control applications for multicomponent robots, in an intuitive and systematic manner.
20 
21 SMACC is inspired by the [SMACH ROS package](http://wiki.ros.org/smach) and it is built on top of [Boost StateChart library](https://www.boost.org/doc/libs/1_53_0/libs/statechart/doc/index.html).
22 
23 Probably the greatest strength of SMACC is that it offers out-of-the-box reference state machines, (found in the folder [sm_reference_library](https://github.com/reelrbtx/SMACC/tree/master/smacc_sm_reference_library)) that you can use, test, hack, and customize to quickly get your application up and running, while also knowing that the library supports advanced functionalities that are practically universal among actual working robots.
24 
25 ## Features
26  * ***Powered by ROS:*** SMACC has been developed specifically to work with ROS. It is a c++ ros package that can be imported from any end-user application package.
27  * ***Written in C++:*** Until now, ROS has lacked a library to develop task-level state machines in C++. Although libraries have been developed in scripting languages such as python, these are unsuitable for real-world industrial evironments where real-time requirements are demanded.
28  * ***Static State Machine Checking:*** SMACC inherits this from the Boost Statechart library which helps the developer to check the consistency of the state machine at compile time (instead of runtime).
29  * ***Component based architecture:*** SMACC has built-in funcionality provided inside SMACC Components that can be dynamically imported at runtime and stored in the local machine. The states only access those components they are concerned with. This enables the SMACC based application to extend or improve the runtime behavior of the system.
30 
31 ## SMACC applications
32 From it's inception, SMACC was written to support the programming of multi-component, complex robots. If your project involves small, solar-powered insect robots, that simply navigate towards a light source, then SMACC might not be the right choice for you. But if you are trying to program a robot with a mobile base, a robotic arm, a gripper, two lidar sensors, a gps transceiver and an imu, then you've come to the right place.
33 
34 ## ROS Integration
35 
36 * ***Intensive use of ROS Action***. SMACC translates Action server events (Result callbacks, Feedback callbacks, etc.) into statechart events. To learn more about this, check the sections Shared Resources and SMACC Architecture.
37 * ***Powerful access to ROS Parameters***. Each SMACC state automatically creates a ros::NodeHandle automatically named according to the SMACC state hierarchy (see more in section Usage Examples - Ros parameters)
38 * ***ROS Navigation built-in funcionality***. SMACC extends the ROS navigation stack in a high level way. It provides specialized navigation planners (for the ROS Navigation Stack) that navigate only using pure spinning motions and straight motions. Implements some mechanism to perform motions recording the path and undoing them later.
39 These can be very useful in some industrial applications where the knowledge or certainty on the environment is higher (ros planners are focused on cluttered and dynamic environments).

46 ## Development methodology
47 SMACC also defines a development methodology where State Machine nodes only contain the task-level logic, that is, the high level behavior of the robot system in some specific application.
48 
49 SMACC applications have low level coupling with other software components of the robot system. SMACC code is recomended to interact with the rest of components the robot system via ROS Action Servers and **Smacc Action Plugins**.
50 
51 The proposed methdology split the states into 2 or more statechart orthogonal lines that comunicate to each other via events. The orthogonal line 0 is tipically for the mobile robot navigation. The second orthogonal line and ahead are used for tools (manipulators, grippers or other custom tools).

57 ## Internal Architecture
58 SMACC State Machines are boost::statechart AsynchronousStateMachines that can work in a multi-threaded application. In SMACC State Machines are two main components that work concurrently in two different threads:
59 
60 * ***Signal Detector***. It is able to handle the action client components communication with action servers and translate them to statechart events
61 * ***State Machine***. It is the end-user code of the state machine itself.

67 # Tutorial
68 SMACC states inherits from boost::statechart:State so that you can learn the full potential of SMACC states also diving in the statechart documentation. However, the following examples briefly show how you create define SMACC states and how you would usually use them.
69 
70 ## Code a minimal SMACC StateMachine
71 In this initial example we will implement a simple state machine with a single state state that executes something at state entry and at state exit. That state machine is described in the following image:
72 
73 <p align="center">
74 <img src="https://raw.githubusercontent.com/brettpac/SMACC/master/documentation/simpleStateMachine.png" width="450"/>
75 </p>
76 
77 SMACC StateMachines and SmaccStates are based on the c++ [Curiously recurring template pattern](https://en.wikipedia.org/wiki/Curiously_recurring_template_pattern) so that the syntax may be strange for some developers but you will notice that it is very easy to follow. The advantage of using this kind of c++ pattern is that the definition of the state machine is correctly written.
78 
79 The following chunk of code shows the minimal SMACC State machine you can create:
80 ```cpp
81 #include <smacc/smacc_state_machine_base.h>
82 
83 struct SimpleStateMachine
84  : public SmaccStateMachineBase<SimpleStateMachine,ToolSimpleState>
85 {
86  using SmaccState::SmaccState;
87  void onEntry()
88  {
89  }
90 };
91 ```
92 
93 Every SMACC State Machine must inherit from SmaccStateMachineBase. The first template parameter is the derived class type (SimpleStateMachine itself) according to the curiosuly recurring template pattern, and the second template parameter (ToolSimpleState) would be the class name of the initial Smacc State
94 
95 ## Code a simple SMACC State
96 
97 For the previous state machine, this would be the initial SMACC State. It also follows the Curiously recurrent template pattern. However, for Smacc states, the second template parameters is the so called "Context", for this simple case, the context is the StateMachine type itself. However, that could also be other State (in a nexted-substate case) or an orthogonal line.
98 
99 ```cpp
100 struct ToolSimpleState
101  : SmaccState<ToolSimpleState, SimpleStateMachine>
102 {
103 public:
104 
105  using SmaccState::SmaccState;
106  void onEntry()
107  {
108  ROS_INFO("Entering ToolSimpleState");
109  }
110 };
111 
112 int main(int argc, char **argv) {
113  // initialize the ros node
114  ros::init(argc, argv, "example1");
115  ros::NodeHandle nh;
116 
117  smacc::run<SimpleStateMachine>();
118 }
119 ```
120 According to the UML statchart standard, things happens essencially when the system enters in the state, when the system exits the state and when some event is triggered. The two first ones are shown in this example. The c++ Constructor code is the place you have to write your "entry code", the destructor is the place you have to write your "exit code". The constructor parameter (my_context) is a reference to the context object (in this case the state machine). This kind of constructor may be verebosy, but is required to implement the rest of SMACC tasks and always follows the same pattern.

122 ## Creating/accessing to SmaccComponents
123 
124 Accessing to SMACC componets resources is one of the most important capabilities that SMACC provides. This example shows how to access to these resources form states.
125 
126 For example, in this case we will asume we are in a state that controls the navigation of the vehicle, and it needs to access to the Ros Navigation Stack Action client and navigate to some position in the environment.
127 
128 <p align="center">
129 <img src="https://raw.githubusercontent.com/brettpac/SMACC/master/documentation/shared_resource.png" width="450"/>
130 </p>
131 
132 The code would be the following:
133 
134 ```cpp
135 struct Navigate : SmaccState<Navigate, SimpleStateMachine>
136 {
137 
138 public:
139  // This is the smacc component (it basically is a wrapper of the ROS Action Client for move base), please check the SMACC
140  // code to see how to implement your own action client
141  smacc::SmaccMoveBaseActionClient *moveBaseClient_;
142 
143  using SmaccState::SmaccState;
144  void onEntry()
145  {
146  ROS_INFO("Entering Navigate");
147 
148  // this substate will need access to the "MoveBase" resource or plugin. In this line
149  // you get the reference to this resource.
150  this->requiresComponent(moveBaseClient_);
151  goToEndPoint();
152  }
153 
154  // auxiliar function that defines the motion that is requested to the move_base action server
155  void goToEndPoint() {
156  geometry_msgs::PoseStamped radialStartPose = createInitialPose();
157 
158  smacc::SmaccMoveBaseActionClient::Goal goal;
159  goal.target_pose.header.stamp = ros::Time::now();
160 
161  goal.target_pose = radialStartPose;
162  goal.target_pose.pose.position.x = 10;
163  goal.target_pose.pose.position.y = 10;
164  goal.target_pose.pose.orientation =
165  tf::createQuaternionMsgFromRollPitchYaw(0, 0, M_PI);
166 
167  moveBaseClient_->sendGoal(goal);
168  }
169 };
170 ```

211 ## Add custom code on Action Result Events
212 
213 In the following example, we want to add some code to the transition between the source state "Navigate" and the
214 destination state "ExecuteToolState". This code may be any desired custom code (for example some transition guard).
215 This code is located in the react method
216 
217 <p align="center">
218 <img src="https://raw.githubusercontent.com/brettpac/SMACC/master/documentation/custom_reaction.png" width="450"/>
219 </p>
220 
221 The following would be the code for this state machine:
222 
223 ```cpp
224 struct Navigate : SmaccState<Navigate, SimpleStateMachine>
225 {
226 public:
227 
228  // With this line we specify that we are going to react to any EvActionResult event
229  // generated by SMACC when the action server provides a response to our request
230  typedef mpl::list<sc::custom_reaction<EvActionResult<SmaccMoveBaseActionClient::Result>>> reactions;
231  using SmaccState::SmaccState;
232  void onEntry()
233  {
234  [...]
235  }
236 
237  // auxiliary function that defines the motion that is requested to the move_base action server
238  void goToEndPoint() {
239  [...]
240 
241  sc::result react(const EvActionResult<SmaccMoveBaseActionClient::Result> &ev)
242  {
243  // we only will react when the result is succeeded
244  if (ev.getResult() == actionlib::SimpleClientGoalState::SUCCEEDED)
245  {
246  // ev.resultMessage provides access to the move_base action server result structure
247 
248  ROS_INFO("Received event to movebase: %s",ev.getResult().toString().c_str());
249  return transit<ExecuteToolState>();
250  }
251  else
252  {
253  return forward_event(); // do nothing, the default behavior if you do not specify any return value
254  }
255  }
256 };
257

259 {
260  using SmaccState::SmaccState;

262  {

268 ## Adding ROS Parameters to Smacc States
269 
270 The SMACC states can be configured from the ros parameter server based on their hierarchy
271 and their class name. It is responsability of the user not to have two different state names at the same level (even if the namespace is distinct since the namespace is trimmed for parameters)
272 
273 For example, imagine a StateMachine to move the mobile robot initially to some initial position, and then moving it to some other position relative to the initial position. You could put the navigation parameters in a ros configuration yaml file like this (and avoid hardcoding):
274 
275 ```yaml
276 MyStateMachine:
277  State1: #Go to some initial position
278  NavigationOrthogonalLine:
279  Navigate:
280  start_position_x: 3
281  start_position_y: 0
282  State2: #Go to some initial position
283  NavigationOrthogonalLine:
284  Navigate:
285  initial_orientation_index: 0 # the initial index of the linear motion (factor of angle_increment_degrees)
286  angle_increment_degree: 90 # the increment of angle between to linear motions
287  linear_trajectories_count: 2 # the number of linear trajectories of the radial
288 ```
289 
290 Then, the c++ code for the State MyStateMachine/State1/NavigationOrthogonalLine/Navigate could contain the following parameter reading funcionality:
291 
292 ```cpp
293 struct Navigate : SmaccState<Navigate, NavigationOrthogonalLine>
294 {
295 public:
296  using SmaccState::SmaccState;
297  void onEntry()
298  {
299  geometry_msgs::Point p;
300  param("start_position_x", p.x, 0);
301  param("start_position_y", p.y, 0);
302  }
303 }
304 ```
305 
306 The param template method reads from the parameters server delegating to the method defined ros::NodeHandle handle does but already located at the exact point in the parameter name hierarchy associated to this state. SMACC is also able have methods getParam and setParam that are delegated to ros::NodeHandle in the same way.
